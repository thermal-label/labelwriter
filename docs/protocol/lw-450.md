# LW 450 Raster Protocol

This page documents the wire protocol of DYMO LabelWriter **300 / 400 /
450 / 4XL** series printers (`engine.protocol === 'lw-450'`). The same
byte stream is accepted by the LabelWriter Wireless / Twin Turbo / SE450
chassis variants. The 550-family chassis uses a different protocol —
see [LW 550](./lw-550).

::: info Source
The byte sequences below are sourced from *DYMO LabelWriter 450 Series
Technical Reference* (Sanford, 2008–2010 revision). The PDF is the
authoritative reference for command bytes; this page summarises the
parts the driver actually emits and parses, with implementation notes
about real-world quirks. The reference document is **not** redistributed
here — DYMO publishes it from their support site.
:::

## Models and engines

Single 300-dpi printer-class interface, one engine per chassis except
the Twin Turbo (two engines, addressed via `ESC q`).

| Family                    | Head dots | Notes |
| ------------------------- | --------: | ----- |
| LW 300 / 310 / 330 / 400 / 450 / SE450 / Wireless | 672 | Standard 56 mm-class head |
| LW 4XL                    | 1248 | 4-inch wide head; same protocol |
| LW Turbo / 450 Turbo      | 672 | Same protocol; firmware optimises feed timing |
| LW Twin Turbo / 450 TT    | 672 | Two engines, addressed via `ESC q '1'` / `'2'` |

`engine.protocol` is `lw-450` for all of the above. The 300-series
firmware is known to reject some commands the encoder never emits for
single-engine devices (`ESC G` short form-feed, `ESC q` select-roll);
those emit only when the device's engine carries a `bind.address`
byte, so the distinction does not surface in practice.

## USB topology

Standard USB Printer-class device. VID is **`0x0922`** (DYMO-CoStar);
PIDs are listed on the [Hardware](../hardware) page.

```
Configuration 1
  Interface 0  —  Printer class  (bInterfaceClass 0x07)
    Endpoint OUT  Bulk    (print data)
    Endpoint IN   Bulk    (status responses — 1 byte per ESC A)
```

Claim Interface 0 directly via `libusb` or WebUSB. There is no
mode-switch step (LabelWriter chassis present as the printer class
from boot).

## Status — `ESC A`

```
1B 41
```

Returns a single byte. Bit layout (active-high — `1` means the named
condition holds):

| Bit | Meaning when set       |
| --: | ---------------------- |
|   0 | Ready (paper in, no jam) |
|   1 | Top of form            |
|   5 | No paper / labels      |
|   6 | Paper jam              |
|   7 | Printer error (catch-all) |

A healthy idle printer returns `0x03` (Ready + Top of form). Bit 0 is
**ready**, not "paper out" — earlier driver revisions had the polarity
inverted, which surfaced spurious "no paper" toasts on healthy units.
Bits 2–4 are reserved.

## Print job structure

A complete 450-protocol job:

```
(once per job)
  ESC @                — reset
  ESC D <bytesPerLine> — set raster row width
  ESC <density>        — set print density
  ESC <mode>           — text vs graphics
  ESC L <len lo> <hi>  — set label length
  [ESC q <roll>]       — select roll (Twin Turbo only)

(per copy)
  <raster rows>        — one row per scan line
  ESC E                — form feed (advance to tear-off)
```

All multi-byte values are little-endian.

### `ESC @` — reset (`1B 40`)

Resets the printer state machine. Always the first command in a job.

### `ESC D <n>` — set bytes per line (`1B 44 nn`)

`n` is the number of payload bytes that follow each raster opcode,
computed as `engine.headDots / 8` — `84` for 672-dot heads, `156` for
the 1248-dot 4XL.

### `ESC <density>` — print density

Single-byte form, no parameter:

| Density  | Byte         | Wire    |
| -------- | ------------ | ------- |
| Light    | `c` (`0x63`) | `1B 63` |
| Medium   | `d` (`0x64`) | `1B 64` |
| Normal   | `e` (`0x65`) | `1B 65` |
| High     | `g` (`0x67`) | `1B 67` |

### `ESC <mode>` — text vs graphics

| Mode     | Byte         | Wire    |
| -------- | ------------ | ------- |
| Text     | `h` (`0x68`) | `1B 68` |
| Graphics | `i` (`0x69`) | `1B 69` |

### `ESC L <lo> <hi>` — set label length (`1B 4C nn nn`)

16-bit little-endian dot count along the feed axis. Example: 200 dots
→ `1B 4C C8 00`.

### `ESC q <roll>` — select roll on Twin Turbo (`1B 71 nn`)

`nn` is one of:

| Byte         | Selection                            |
| ------------ | ------------------------------------ |
| `'0'` (`0x30`) | Auto — firmware picks an available roll |
| `'1'` (`0x31`) | Left roll                            |
| `'2'` (`0x32`) | Right roll                           |

Twin Turbo engine entries store `0x31` / `0x32` directly in
`engine.bind.address`; the encoder hands the byte through unchanged.

### Raster row — `0x16` (uncompressed)

```
16 <bytesPerLine bytes>
```

`0x16` is the SYN opcode. The payload is exactly `bytesPerLine` bytes
(NOT length-prefixed — the prior `ESC D` set the row width). Each bit
is one dot; bit 7 of byte 0 is the leftmost dot at the start of the
head.

### Raster row — `0x17` (RLE compressed)

```
17 <bytesPerLine bytes of run-length pairs>
```

PackBits-style run-length: each output byte encodes a 1-bit run of
length `1..128`. Top bit is the bit value; the low 7 bits are
`run_length - 1`. The total decoded length is exactly `bytesPerLine`
bytes.

The driver enables compression only when `options.compress: true`. On
USB the uncompressed path is faster; on slower transports compressed
rows shrink wire time.

### `ESC E` — form feed (`1B 45`)

Advances the printed label to the tear-off position. Sent once per
copy. The driver concatenates `<rasterBlock> <ESC E>` for each
requested copy in one job.

### `ESC G` — short form feed (`1B 47`)

Advances by a small fixed amount. Used between copies when an
explicit short feed is wanted; the encoder does not emit it for the
canonical multi-copy job (which uses repeated `<rasters> + ESC E`).
The 300-series firmware rejects this opcode — the encoder only emits
it on devices whose engine declares it, so the rejection does not
surface.

## Error recovery

To clear a stuck command buffer and re-sync the parser, send 85 `ESC`
bytes followed by `ESC ESC A`:

```
[1B × 85] 1B 1B 41
```

`buildErrorRecovery()` in the encoder produces this sequence. The
printer responds with a fresh status byte once the parser is back at
top-of-frame.

## WebUSB

Same protocol as USB; the browser package opens Interface 0,
`transferOut` for the encoded job, `transferIn` for the status reply.
WebUSB requires HTTPS or `localhost`.

## Porting checklist

- [ ] Use VID `0x0922` and the per-model PIDs from
      [Hardware](../hardware); claim Interface 0.
- [ ] Begin every job with `ESC @`, then `ESC D <n>`, then density and
      mode.
- [ ] Use `ESC L <lo> <hi>` to declare the label length in dots
      (little-endian).
- [ ] Each raster row is exactly `bytesPerLine` bytes preceded by
      `0x16` (or `0x17` with RLE pairs); never length-prefixed
      separately.
- [ ] End each copy with `ESC E`; for multiple copies repeat
      `<rasters> + ESC E`.
- [ ] On Twin Turbo, emit `ESC q '0' / '1' / '2'` after the per-job
      preamble to pick a roll.
- [ ] Read 1 byte after `ESC A`; bit 0 is **ready**, bit 5 is
      **no paper**, bit 6 is jam.
- [ ] Don't send `ESC G` to 300-series firmware — it rejects the
      opcode. The encoder gates this on engine bind/address presence.
